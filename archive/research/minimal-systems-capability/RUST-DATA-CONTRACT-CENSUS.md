# Rust 1.97.0 Data-Contract Census

Status: manual semantic coverage and obligation clustering, 2026-07-14. This census selects
no xlang syntax, storage representation, privileged transition, standard
container, optimizer fact, or production mechanism.

## Scope and provenance

`RUST-DATA-CONTRACT-CENSUS.tsv` groups the selected stable Rust 1.97.0
data-structure surface into coarse caller-observable coverage and obligation
clusters. The source snapshot is
the official `1.97.0` tag, annotated tag object
`eca4cdea45792600b4275e9d4c64fd827d575a24`, peeled commit
`2d8144b7880597b6e6d3dfd63a9a9efae3f533d3`, and tree
`90d170433978b46bb035bb89f8a7bfa41c5d665f`. Exact raw declarations and source
anchors remain in `RUST-1.97.0-API-INVENTORY.tsv`; the generated inventory and
module digests remain pinned by `RUST-1.97.0-CENSUS-MANIFEST.json`.

Concrete trait implementations are a separate evidence channel because the
declaration inventory intentionally does not repeat them on every implementer.
`RUST-1.97.0-TRAIT-IMPL-CROSSWALK.tsv` contains the exact 334 selected direct
implementations used by the cross-cutting trait clusters and the new Range
iterator rows. Its builder records associated bindings, method/result shapes,
stability, rustdoc identities, exact rust-src line identities, and source-line
SHA-256 values. These rows do not increase any public-declaration or coverage-
cluster count.
`G0-TRAIT-IMPL-TOPOLOGY-ROUTING.tsv` separately gives every implementation key
one exact primary refinement family or gate plus predecessor-family,
predecessor-gate, and implicated/reopening targets under a closed classifier.
That routing establishes applicability only; it does not turn a concrete
implementation into an exact xlang member contract or a closed family result.
For the four cross-family trait-operation clusters, topology alone is not the
complete applicability authority. Evidence-universe policy v3 composes each

two distinct applicable targets and 281 have one; every additional operation
gate retains the full route-level prerequisite set and also records the topology

`RUST-DATA-SURFACE-MAP.tsv` is the exact declaration crosswalk. It contains one
row for each canonical stable-safe seed declaration, keyed by the raw

`RUST-D10-SURFACE-MAP.tsv` is a second exact, mechanically generated crosswalk

## Coverage-clustering and later-normalization rules

Each row conservatively summarizes demand using the charter shape:

```text
pre-state + input ownership + behavior parameters
    -> post-state + result ownership + failure/destruction effects
```

Rust-surface brace notation is exhaustive shorthand: for example,
`Vec::{pop,remove}` expands to both declarations, and
`{btree_map,hash_map}::OccupiedEntry::{get,remove}` expands over both helper
families. Every one of the 545 canonical stable-safe seed declarations appears
in a primary family row or a separately marked raw-obligation row. The 35
stable-unsafe declarations appear only in `RAW-UNSAFE-*` evidence rows.
Cross-cutting `TRAIT-*` and `ALLOC-*` rows summarize protocols already used by
family rows; they do not create a second primary classification.

The D10 crosswalk applies the same coverage rule to stable iterator,
`core::range`, and legacy `core::ops` range surfaces. Public adapter/helper
struct names and namespace reexports are redundant spellings. Range values,
borrow/clone/map operations on bounds, range-bound protocols, mutable legacy
cursor state, membership tests, emptiness tests, endpoint access, and iteration
are separated into 29 coarse clusters because bound inclusion, comparison
direction and count, equality behavior, finiteness, exhausted-state behavior,
overflow, and exact-size/fused promises are useful G0 obligation boundaries.
This separation is not a claim that each cluster is one exact operation
contract.

The verifier fails on any omitted or repeated canonical declaration, extra
mapping, changed source identity, unknown contract ID or marker, empty required
field, cross-cutting row used as a primary classification, or unaccounted
stable-unsafe seed declaration. It also pins the D10 132/43 domain split, the
138 cluster and 37 redundant-surface routes, the 29 distinct range clusters,
the independence of `ExactSizeIterator` and `FusedIterator`, the exact six-way
payload partition, all 294 branch keys and role counts, and the
`scope_owner_contract_ids` edge for every delegated, boundary, and frame row.
It rejects unrestricted `E` or `P` for every deferred, delegated,
boundary-evidence, or frame-deferred contract.

The concrete-implementation verifier independently pins all 334 implementation
identities, exact family and trait counts, associated bindings and result
shapes, source-line hashes, and owning-cluster coverage. Three compiled Rust
1.97 canaries prove that all 21 stable-reachable Step endpoint types admit each
new Range iterator entrance, while stable Rust rejects both the unstable
`core::ascii::Char` endpoint and a downstream custom Step implementation with
exact E0658 diagnostic digests and predicates.
The topology-routing verifier independently pins the exact classifier classes,
class targets, per-class membership digests, ordered route digest, and complete
one-to-one join to those 334 implementation identities. It rejects fuzzy,
prefix, regex, default, unknown, or unmatched implementer routing and keeps
family predecessors distinct from evidence applicability.

G0 clustering is not semantic equality. It preserves enough distinctions to
bound later work, including:

- trapping and recoverable access or reservation;
- shared, unique, and owning traversal;
- ordered removal and unordered swap removal;
- stable sort, unstable sort, and nth selection;
- eager mutation and lazy cursors with abandonment behavior;
- sorted, heap-internal, sequence, and arbitrary hash iteration order;
- deep clone and shared-identity clone; and
- stable contract bounds, conditional expected bounds, and explicitly current
  Rust implementation costs.

Before candidate construction, a scored experiment, or any family-level `E` or
`P` claim, the applicable Family Lock must derive its exact cluster set from
the routing registry and account for every applicable evidence identity and
B/M/W/H/O obligation in the two generated universe registries. It must assign a stable
`member_contract_id` to each distinct member contract and an `outcome_id` to
each distinct normal, recoverable-failure, checked-trap, abandonment, and
destruction outcome. Each exact unit must freeze ownership, result provenance,
invalidation, cleanup, behavior effects and call counts, allocation behavior,
asymptotic and structural costs, capability IDs, payload-role keys, algorithms,
canaries, and measurement endpoints. Two declarations may share a member ID
only after the lock proves equality across all of those dimensions. A shared
cluster ID or similar spelling is not that proof.

Two ownership-sensitive boundaries are pinned mechanically. Strict
`String::from_utf8`, borrowed-or-owned `String::from_utf8_lossy`, and
`FromUtf8Error` access are three contracts: only the lossy valid branch borrows
the input, while `FromUtf8Error::as_bytes` borrows the error owner and
`into_bytes` consumes that owner. Map-entry provenance is likewise
branch-specific: an occupied key borrow derives from map storage, a vacant key
borrow derives from the candidate owned by the guard, and a successful insert
changes subsequent result provenance to map storage.

The same source adjudication distinguishes retained payload from cursor-only
authority. A vacant entry guard owns its candidate key and must transfer,
return, or destroy it exactly once. An occupied entry guard owns no arbitrary
key or value payload; its outer `key`, `get`, `get_mut`, and `into_mut`
projections are already accounted as map-rooted result borrows. Likewise,
`Rc<MaybeUninit<T>>` constructors contain no live `T`, so they create no
stored-borrow payload branch until a later safe fill/seal transition establishes
a live `T`.

`implementation_privilege_evidence` records what Rust uses or exposes; it is
not a proposed xlang mechanism. In particular, safe Rust surfaces may rely on
raw relocation, private uninitialized or sparse state, guard `Drop`, and unsafe
reconstruction. A safe call that exports a pointer, `MaybeUninit`, or a leak
obligation is classified separately from an ordinary safe-library derivation.

The failure/destruction column preserves Rust panic-unwind behavior where that
behavior is part of the source contract or implementation pressure. It is not
silently imported into xlang. The derivation matrix translates a Rust unwind
edge through current EFF-4: an xlang trap aborts without cleanup or a
recoverable post-state, while still forbidding an invalid read, double drop, or
other undefined behavior before abort. Normal callback errors, early returns,
and other non-trap exits retain their exact ownership and destruction duties.

## Current-xlang status

The status vocabulary is inherited from the owner-reviewed capability report:

- **E** — directly expressible now with the required contract and no known
  structural tax;
- **P** — established blessed pattern with correctness and performance
  evidence;
- **U** — plausible workaround that still lacks a complete proof or cost
  result; and
- **X** — current gap, asymptotic loss, forbidden privilege, or necessarily
  pathological simulation.

Statuses are deliberately cluster-local screens. They describe whether the
whole coarse envelope has a plausible current route; they do not authorize an
exact member contract, experiment, or family closure. Current xlang directly supports
checked metadata/index operations and some in-place operations for fixed,
fully initialized Copy buffers; that does not promote a growable affine
sequence contract. The measured append-only SoA pool remains a protected P
baseline, not evidence for deletion, recycling, shared identity, or keyed
storage. Validated UTF-8 ownership, general affine containers, lazy cleanup
guards, shared ownership, dynamic borrowing, and safe ordinary-library
occupancy transitions remain X or explicitly staged outside the current
language.

## Load-bearing findings

1. Lazy `drain`, `splice`, extraction, mutable heap peek, and dynamic-borrow
   guards are cleanup protocols, not iterator convenience. Rust repairs private
   transient state in `Drop`; xlang cannot rely on a writer remembering a
   `finish` call or on an affine token that may be abandoned.
2. Rust's stable recoverable allocation surface is incomplete. Ten stable
   `try_reserve` methods cover six families, but `TryReserveError::kind` remains
   unstable, most fallible constructors remain unstable, and ordinary
   allocating methods may enter the divergent OOM handler. Capacity overflow,
   allocator failure, OOM, input ownership, and rollback therefore need an
   independent xlang contract.
3. Sparse metadata and dynamic borrow flags are optimizer fact channels. A
   control byte or borrow flag authorizes payload access only while every
   mutation and invalidation rule preserves the relation. Green functional
   tests are not hostile fact review.
4. Whole-place `swap`, `replace`, and `take` are foundational ownership
   transitions. Rust's `replace` implementation explicitly avoids the extra
   movement of a swap-based simulation; these rows must not collapse into Copy
   assignment.
5. Rust API presence is demand evidence, not a completeness or mechanism
   oracle. Rust std omits generational pools, graphs, LRU caches, indexed
   priority queues, inline-small sequences, ropes, and other required
   generativity witnesses, while its safe library implementation has raw and
   destructor privileges unavailable to an ordinary xlang library.
6. Stable iteration is a composition contract, not just loop syntax. Lazy
   producers, adapters, and consumers must compose as direct monomorphized
   cursor state without an intermediate collection or per-item allocation;
   stateful callbacks, short-circuit resumption, affine-item abandonment, and
   multi-output partial construction remain observable obligations.
7. Iterator metadata is not payload-validity authority. `size_hint` is
   advisory, stable `ExactSizeIterator` supplies an exact representable count
   but is not Rust's unsafe `TrustedLen`, and `FusedIterator` promises only that
   no item reappears after `None`. These promises remain distinct fact channels.
8. New `core::range::iter(&self)` is not a stable generic behavior contract.
   In the pinned Rust 1.97.0 source, `Step` and `TrustedStep` are unstable, so
   stable downstream code cannot name or implement them. The exact stable-callable
   canonical set is 21 borrow-free `Copy` types: all twelve signed and unsigned
   integers, `char`, `NonZero<u8/u16/u32/u64/u128/usize>`, `Ipv4Addr`, and
   `Ipv6Addr`; the additional `AsciiChar` implementation is excluded because the
   type itself is unstable. Cursor creation copies endpoint values, retains no
   descriptor borrow, and preserves the descriptor because these concrete source
   values are `Copy`, not because `&self` implies purity. The external demand is
   therefore sealed monomorphized checked stepping with no public or unsafe `Step`
   authority, user `Clone`, generic callback, or behavior-effect relation.
   Half-open and inclusive stepping checks order before successor computation;
   inclusive iteration yields the maximum endpoint once without computing
   `max + 1`. The unbounded route yields its maximum once, then xlang's next
   demanded step traps before cursor mutation and never wraps. The source anchor
   is Rust commit `2d8144b7880597b6e6d3dfd63a9a9efae3f533d3`, with
   `library/core/src/iter/range.rs` SHA-256
   `ae3f9307f4b4972f418561ae2a0311586eb3dde782359b8aaef3244915256464`.
9. Receiver-region bounds and physical provenance roots are distinct.
   `RangeBounds` over owned endpoints projects receiver fields, while
   `Range<&T>` and tuple `Bound<&T>` implementations return a shortened
   reference whose physical root remains external. Direct borrowed-empty
   `Default` values similarly require valid alignment and an exact zero-length,
   empty access footprint. Independently produced mutable empties may have equal
   pointer values and still coexist; this grants neither pointer-inequality
   authority nor `BR-DISJOINT`.

The TSV is a G0-Core accounting input. A cluster's demand can be claimed closed
only after the applicable Family Lock maps every declaration, implementation,
and selector child in the complete applicable evidence universe to exact
`member_contract_id` and `outcome_id` units and its
derivation ledger supplies an ordinary checked-library witness, normal-exit and
abandonment proof, asymptotic and structural cost account, fact and invalidation
ledger, negative soundness canaries, and exact family-lock evidence. Nothing in
this census authorizes that later work.
