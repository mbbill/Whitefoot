# Privileged Basis Research Verdict

Date: 2026-07-15

Status: owner packet. Proposed D14 architecture pending owner review. Exact
dense D-2 and exact P-1 remain pending. No production action is authorized.

## 1. Proposed selection

The research recommends one privilege boundary and two semantic planes.

### One nonforgeable gate

Use one sealed stateful admission verifier per authenticated semantic toolchain
root. Compiler-embedded genesis entries and signed successor-snapshot entries
pass the same exact use-time predicate. Authorization binds the public contract,
semantic and implementation dependency cones, consumer cone, target/backend,
frame and program scope, current active membership, review identity, and exact
final resolved implementation.

No source keyword, attribute, module path, package identity, helper name, build
flag, environment value, plugin, linker alias, or cache record grants privilege.
`CERTIFIED` exact bytes and an explicitly attenuated `TRUSTED_FOREIGN` provider
remain distinct outcomes. Replacing the trusted compiler or semantic root is a
TCB replacement, not an in-language second gate.

### One checked public resource basis

Above the gate, ordinary unapproved modules may carry producer-generated proofs
checked by one fixed public reference policy. Solvers, AI generation, proof
search, optimizers, and derived proof rules remain untrusted. A derived rule
must prove implication to the fixed policy and cannot install an axiom.

The candidate basis has six independent authority axes:

1. spatial resource and checked typed-place authority;
2. logical place identity and stale-reuse prevention;
3. lifecycle, exact transition scope, and executable disposition;
4. interference and weak-memory authority;
5. external frame transitions; and
6. exact final loaded-code binding.

Its common storage vocabulary is a generative byte carrier partitioned into
checked typed places and padding, explicit vacant/full state, linear move and
release authority, exact root/epoch/offset identity, exact focus, proof-only
partition reshape, source-preserving structural `CoreCopy`, checked callables
and code leases, and executable static disposition.

Named containers, a privileged standard library, a universal topology tag, a
universal liveness bitmap, a universal destructor pointer, and writer-visible
unchecked code are not part of the proposal.

## 2. Why this candidate is preferred

The proposal is minimal by independent semantic authority rather than by
operation-name count. Combining two obligations behind one name does not make
either obligation disappear. Eighteen removal witnesses separate the retained
authorities; removing any one breaks safety, exact lifecycle, ordinary-library
generativity, final-code binding, or a protected structural-cost hypothesis.

Category-level paper witnesses cover all 26 systems-demand categories. They
include dense and sparse storage, ring and gap layouts, recursive and arena
ownership, stable recyclable identity, shared and dynamic borrowing,
concurrency and reclamation, pinning, partial I/O, callbacks, async
cancellation, persistence, devices, and loaded code. Four independently chosen
held-outs exercise concurrent cuckoo migration, crash-consistent LSM state,
hot-swappable JIT code, and sparse GPU residency. No listed witness found an
architecture-shape need for a named privileged container or a second gate.

The literature and implementation precedents support the components rather
than the complete xlang combination: foundational proof-carrying code and
untrusted proof production support the fixed-checker boundary; permission and
separation logics support generative resources, borrowing, and interference;
RefinedC and SteelCore support foundational checked library proofs; GhostCell
supports generative identity; and verified allocator work supports ordinary
proof-carrying resource modules. None proves xlang's complete no-unsafe,
no-GC, no-unwind, ordinary-library, predictable-AI, protected-performance goal,
so the project-specific exact gates remain necessary.

a syntax decision, or proof that the current compiler implements the basis.

## 5. Owner decisions requested

### Decision A: research architecture

Should the one sealed stateful gate plus one fixed checked public
resource-policy architecture be selected for further research only?

Recommendation: **yes**. No architecture-shape blocker was found, the retained
authorities have separating witnesses, and the known failures are exact source
partition and schedule evidence rather than a demonstrated missing primitive.
Ratification must keep exact D-2 and P-1 pending.

### Decision B: bounded normalization pass

Should the project authorize one bounded research pass that:

- splits coarse Convert direction and callable contracts;
- freezes exact outer-allocator event schedules;
- splits positive-size/full, relocating, included-ZST, and deferred-ZST
  IntoOwner/IntoBoxed contracts;
- reconstructs all seven trace-classifier schedules, including checked-
  arithmetic reach before behavior abort, growth before behavior, unstable
  select/sort exchange and permutation, and stable-sort relocation and scratch
  acquisition; and
- reconstructs exact same-contract Rotate and stable/cached-key sort schedules?

Recommendation: **yes**. This pass should perform mechanical source
normalization and reference-trace reconstruction first, then return only genuine
semantic or performance choices to the owner. It must not change the language,
specification, compiler, basis, or production code, and must not run benchmarks
without separate authorization.

No syntax, proof-language, verifier implementation, weak-memory profile,
surface keyword, standard library, container implementation, E0.1 restart,
compiler migration, benchmark, or default-teaching decision is requested now.
