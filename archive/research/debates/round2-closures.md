# Round 2 — Closures / higher-order functions (`closures`)

**Verdict:** `replace_with_alternative`

## Rationale

The null hypothesis stands, but only because the alternative is promoted to a language requirement. On the evidence: the closures/captures coverage row rests on a single card, D009, at low_to_medium confidence — and D009 is an optimizer-defeat card; the llvm-mlir capability map's closure-capture row is all TBD; no F/A/C/J card attests any benefit of capture semantics; and decision-gates.md bars admitting features on optimizer-attractiveness alone, a bar closures do not even reach. Both design_candidate_later leaders are structurally hostile: the dispatch leader admits exactly one implicit path (guaranteed static specialization) plus an explicit restricted dyn construct, and the aliasing leader has no carded callback story at all (round1-aliasing-model line 52; misleading-comparisons #2). The defender's case is the strongest possible for the feature and it still fails at its load-bearing joint: the claim that the alternative has only indirect encodings (D002/D007/D008 cost class) or tag-enum apply loops is refuted by composition with the corpus's own leading generics candidate — an explicit env struct passed as a monomorphized type parameter is machine-code-isomorphic to the defender's 'closures as unique anonymous types consumed by specializing generics.' The defended restricted form IS compiler-performed defunctionalization; since the optimizer cannot distinguish them, optimizer necessity is not proven, and the defense's real content is (a) requirements the alternative must satisfy (guaranteed direct calls via the generic encoding, checked capture-mode declaration, non-escaping guarantees by construction rather than F007/J004-conditional recovery) and (b) a verification residue (bidirectional exactness, by-name binding) plus EMPIRICAL-UNTESTED drift-rate claims that cannot overcome the null hypothesis but do define the reopening experiment. The defender's killer snapshot-vs-live instance is symmetric once env-field types are recognized as declared capture modes under the ownership core; the prosecutor's 'every drift mode is a compile error' is likewise overstated (population-site wrong-referent and field-type choice type-check), which is exactly why this is replace_with_alternative rather than cut_from_core: a bare cut leaving fn-pointer+env as the idiomatic encoding would reinstate D009's defeat shape everywhere, and a cut without mandated exactness diagnostics would hand a fallible-at-scale writer the three-artifact drift channels unchecked. Cutting deletes real obligations — capture inference, closure conversion as a verified-IR proof burden, the TBD lowering row, and the F004-flagged anonymous-escaping-environment complication to the memory-object model — while the escaping/heterogeneous residue lands in the already-planned explicit dynamic construct, where the defender itself concedes closed tag-enums serve optimizer and verifier better than erased function values. This verdict is evidence-conditional per the standing rule: every load-bearing D-card is a provisional direct extraction, and the named experiment can reopen the surface (not the semantics) as a checked desugaring layer.

## Optimizer role

Net negative-to-neutral, and therefore insufficient for core inclusion. The corpus's only direct card on function values is a defeat card: D009 documents Function-typed fields hiding representation/call targets and forcing dynamic dispatch even inside fully closed programs — the round1-dispatch-generics judge used it against the closed-world advocate, and line 52 of that verdict records first-class-function dispatch as a gap no position covered. General closures are implicit members of D002's indirect-call cost class (lookup cost, prevents inlining), recoverable only via D007's conditional/speculative/revertible devirtualization, blocked by D008 interposition, with environment allocation recoverable only via F007/J004's conditional (explicitly not guaranteed) escape analysis, and with J001/J002 showing the profiling/guard/deopt machinery a runtime needs once call targets are values. The defender's counter-asymmetry (guaranteed direct calls via closures-as-unique-static-types) is real but not distinctive: its 'exactly two realizable encodings' claim for the alternative is FALSE given the dispatch-generics leading candidate — an explicit env struct passed as a type parameter implementing a call interface, monomorphized per D001, yields the identical guaranteed-direct-call, exact-layout machine code. The restricted closure IS compiler-performed defunctionalization; the optimizer sees no difference. So closures add no optimizer capability the mandated alternative lacks, while unrestricted closures add D009-class defeats and an all-TBD verified-lowering obligation (llvm-mlir-capability-map closure-capture row). The defender's minimal-footprint argument (capture lists feeding F001/F002/F003 facts) is partially valid — exact capture lists guarantee footprint minimality where env structs make it a convention — but footprint SOUNDNESS is equal in both designs under the checked-ownership aliasing leader, minimality-payoff magnitude is uncarded, and misleading-comparisons #2 flags callback lowering to noalias as unresolved for every design. Per decision-gates.md, even optimizer-attractiveness would not suffice for a default; closures do not clear the lower bar.

## Verification role

This is the genuine contest, and closures' one real contribution must be reconstructed in the alternative rather than used to keep capture semantics. The defender's strongest point: mandatory explicit capture lists give a bidirectional exactness check (no unlisted free variable, no unused capture) plus by-name binding at the creation site, whereas the manual triple (struct definition, population site, function body) has no native correspondence check — a stale or over-wide env struct type-checks. But the defender's killer instance, snapshot-vs-live, is SYMMETRIC on inspection: under the ownership-based aliasing leader, an env-struct field's declared type (owned T vs &T vs &mut T) carries exactly the same declared, compiler-checked capture-mode information as an explicit capture-mode annotation; the wrong choice type-checks in both surfaces, so 'the intent is no longer declared anywhere' is overstated. The recoverable residue is therefore: (1) exactness — must become mandated core ERRORS over env structs (unused field, uninitialized field, dead dispatch variant, non-exhaustive dispatch), not lints, as the prosecutor conceded; (2) by-name binding — population-site wrong-referent wiring (.x = wrong_but_type-compatible_var) remains checkable only by type, not by name, in the alternative; this is the one verification property that cannot be fully reconstructed and is the strongest surviving argument for a later checked-desugaring layer, pending the named experiment. Cutting the feature also DELETES verification obligations: capture inference, closure conversion as a semantics-preservation proof burden in the verifier-checked IR strategy, and the entirely-TBD capability-map row; F004's negative scope (lifetime/provenance/escape rules precede strong allocation facts) says anonymous escaping environments are precisely the hard case the memory-object model would otherwise have to settle first. C004 Send/Sync checks apply to env structs as ordinary structs with no new machinery.

## AI-writer considerations

All AI-writer behavioral claims on both sides are EMPIRICAL-UNTESTED — the corpus contains zero AI-codegen evidence (caveats.md). Ledger as judged: (1) Token/context cost — the alternative loses: each function value becomes struct + top-level fn + construction wiring with globally unique names, and re-reading a call site requires pulling nonlocal definitions into context; conceded, admissible under the charter, magnitude unmeasured. (2) Consistency drift — the alternative's most serious cost, but the mandated diagnostics (see requirements) convert missing-field/arity/stale-shape drift into machine-readable compile errors; the residual silent channels are population-site wrong-referent (type-compatible but wrong variable) and snapshot-vs-live field-type choice — the latter symmetric with capture-mode choice under closures, the former genuinely worse without by-name binding. (3) Edit amplification — the defender's central-enum objection is valid against tag-enum defunctionalization but not against the mandated generic-parameter encoding, which is the required hot-path form; tag enums are confined to genuinely heterogeneous stored-callback sets, where the defender itself concedes closed enums beat dyn function values for optimizer and verifier alike. (4) Diagnostics — env-struct errors are ordinary struct/ownership errors the core must emit anyway; capture diagnostics would entangle with an aliasing model that has no carded callback story (round1-aliasing-model line 52). (5) Distribution shift — AI writers trained on closure-rich corpora may emit lambda-shaped code or defunctionalize incorrectly; symmetric risk (wrong captures that type-check in a capture-bearing core) and untested in both directions. Compile-time in the iteration loop: unmeasured, plausibly a wash (more source volume vs deleted capture-inference/closure-conversion/escape-analysis phases).

## What must be true

- The language MUST provide the guaranteed-direct-call encoding: env structs passable as type parameters implementing a call interface, monomorphized under the dispatch-generics leading candidate (D001), so parameterized-algorithm callbacks are never forced through fn-pointer+env indirect calls; the dispatch leader must itself survive its verification program (F003-discharge check, code-size measurement).
- The language MUST mandate machine-readable core ERRORS (not lints) reconstructing capture-list exactness over the manual artifacts: unused env-struct field, field used but never initialized, dead dispatch variant, non-exhaustive tag dispatch, signature mismatch between env struct and consuming function — with diagnostics that name all three artifacts (struct, function, construction site) so the AI repair loop is single-hop.
- The language MUST make the snapshot-vs-live choice explicit and coercion-free: env-struct field types (owned vs shared-borrow vs exclusive-borrow) are the declared capture modes, checked by the ownership core (C004/F001 machinery), with no implicit copy/borrow coercion at population sites.
- Escaping and heterogeneous function values route ONLY through the explicit, restricted D003-style dynamic construct (or closed tag-enums with exhaustiveness checking); Function-typed fields with hidden call targets (D009's defeat shape) must be inexpressible in the core.
- Structured-array body regions (A002/A006/A010 surface) must be specified as second-class, non-escaping, capture-free-or-explicitly-parameterized regions, with the closure/region line drawn precisely so later rounds cannot smuggle first-class capture back in through loop bodies.
- The toolchain must enforce single-definition discipline for repeated callback patterns (dedup detection of structurally identical env-struct/function pairs across modules) to contain cross-module duplication drift.
- The aliasing model must acquire a carded story for borrows stored in env structs lowering to F003 noalias/alias.scope facts — misleading-comparisons #2 names callbacks as the fact-destroyer, and this verdict inherits that unresolved obligation.
- The load-bearing provisional cards D001, D002, D007, D009 must survive adversarial verification (caveats.md standing rule); if D009 or the D007-conditionality asymmetry falls, the cost side of this ruling must be re-weighed.
- The AI-writer drift experiment (see follow-ups) must not show a dominant silent-semantic-divergence rate for mandated-diagnostic defunctionalization versus checked capture-list closures; if it does, a capture-list desugaring layer that compiles to exactly the mandated encoding should be added — reversing the surface decision without touching this ruling's core semantics.
