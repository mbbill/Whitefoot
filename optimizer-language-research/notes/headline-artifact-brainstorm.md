# Headline Artifact Brainstorm (2026-07-10)

5-lens panel (opus), archived from workflow wf_d0b9bf49-ba1.

## Lens: security-press

### zlib-drop: ABI-compatible inflate (libz.so.1)  `[near]`
- **Headline:** An AI wrote a drop-in replacement for zlib's decompressor that is bit-identical, C-fast, and in which the CVE-2022-37434 heap-overflow class cannot be written.
- **Vs Rust wave:** Rust already has zlib-rs (now on Chromium/Firefox paths) and miniz_oxide, but both carry unsafe blocks in their SIMD and pointer hot loops, so the buffer-arithmetic bug class is only mostly, not provably, gone; Whitefoot has no unsafe hatch at all, so the over-read is unrepresentable including on the fast path, and zlib-rs was human-written whereas this is AI-authored end to end.
- **Swap-in:** LD_PRELOAD or symlink over libz.so.1 exporting inflateInit2_/inflate/inflateEnd/uncompress with byte-identical semantics; nearly every dynamically-linked Linux program picks it up with no relink. Gold-standard drop-in of the whole slate.
- **Verify:** Bit-identical output over Canterbury/Silesia plus a large real-world blob corpus; pass zlib's own test suite; libFuzzer fuzz-diff over millions of inputs asserting identical bytes or identical error code, including replay of the CVE-2022-37434 gzip-extra-field PoC.
- **Gaps:** Growable/streaming output buffers (the one near-term gap); the inflate window/state machine expressed over byte buffers + structs + enums (have); Huffman tables (near const tables). No floats/strings/concurrency needed.
- **Risk:** Matching zlib's exact streaming/flush and error-code semantics bit-for-bit is fiddly, and zlib-rs already sets a high safe-language speed bar, so performance parity on the Huffman inner loop must be demonstrated, not asserted.

### safe-webp: VP8L lossless decoder (the CVE-2023-4863 kernel)  `[near]`
- **Headline:** The libwebp heap overflow behind Apple's BlastPass and the Chrome WebP 0-day (CVE-2023-4863) is structurally impossible in this AI-written decoder.
- **Vs Rust wave:** image-webp (Rust) was written partly in response to this very CVE, so Rust did move here; the differentiator is that the exact defect (an out-of-bounds write while building the VP8L Huffman table) is a checker-proven impossibility rather than a bug that merely wasn't reintroduced, the fast Huffman path needs no unsafe, and it is AI-authored.
- **Swap-in:** LD_PRELOAD over libwebp exporting WebPDecodeRGBA/WebPGetInfo for the lossless path; drops into Chromium/Electron/Android image pipelines and any app that vendored libwebp (which in 2023 was essentially everything).
- **Verify:** Decode libwebp's lossless conformance suite bit-identical to WebPDecodeRGBA; run the public CVE-2023-4863 PoC file and confirm a clean reject; fuzz-diff RGBA output against stock libwebp.
- **Gaps:** Color-cache, predictor/subtract-green/color transforms, and Huffman group tables are all integer/table work (have + near const tables). Full WebP also needs the VP8 lossy path (integer IDCT), deliberately out of first scope.
- **Risk:** VP8L has several interacting transforms and getting all bit-identical is real work; lossy VP8 (the more common variant on the web) is a much larger follow-on, so a lossless-only shim is a partial swap-in until that lands.

### safe-mp4: ISO-BMFF/MP4 box demuxer  `[now]`
- **Headline:** The Stagefright class of MP4 integer-overflow-to-heap-overflow bugs that let an MMS root your phone cannot be expressed in this AI-written demuxer.
- **Vs Rust wave:** Mozilla already wrote mp4parse-rust to de-risk exactly this in Firefox, so Rust took the container-parser story; but that library still uses unsafe at the FFI boundary and was human-written, whereas Whitefoot's is AI-authored with zero unsafe, and the size/offset integer overflows that defined Stagefright are excluded by checked arithmetic (no silent wrap) rather than by hand-audited bounds checks.
- **Swap-in:** Ships as a standalone C-ABI libmp4parse-style validating demuxer (box tree + sample tables), or as a safe pre-parser sidecar that validates/normalizes a file before an untrusted C demuxer sees it. Weaker than a single universal ABI because MP4 demux APIs are app-specific.
- **Verify:** Fuzz-diff the extracted box tree and sample-to-chunk/offset tables against ffmpeg/GPAC over a large media corpus; replay the archived Stagefright PoC files (CVE-2015-1538 family) and confirm clean rejects.
- **Gaps:** A fixed-capacity streaming validator is genuinely today's subset (bounded integer offset/length parsing over byte buffers, structs, enums, recursive box tree). Full variable-length sample-table extraction wants growable buffers (near). No floats (fixed-point matrices are copied, not computed), no strings, no concurrency.
- **Risk:** The box zoo is broad (edit lists, fragmented MP4, dozens of box types), so full coverage is tedious even though demux-only avoids codec work; and there is no single canonical ABI to preload, so swap-in is via an embedded library or sidecar.

### safe-png: libpng-ABI decoder  `[near]`
- **Headline:** An AI reimplemented PNG decoding as a libpng drop-in, bit-identical on PngSuite, with the CVE-2015-8126/CVE-2018-13785 overflow class unrepresentable.
- **Vs Rust wave:** image-rs/png exists and is good, but it is a crate for Rust programs, not a libpng ABI drop-in, so the enormous installed base of C/C++ apps linking libpng16.so cannot adopt it; Whitefoot emits the C ABI, keeps zero unsafe through the filter inner loops, and is AI-authored.
- **Swap-in:** LD_PRELOAD/symlink over libpng16.so for the common read path (png_read_png/png_read_image). Partial, because libpng's callback-and-struct API surface is large, so the realistic first target is the high-level read entry points.
- **Verify:** Decode the official PngSuite images bit-identical; run libpng's pngtest; fuzz-diff against libpng and replay known-crash PNGs (interlaced and oversized-chunk PoCs).
- **Gaps:** Depends on the inflate kernel (idea 1) plus PNG filters (integer), Adam7 de-interlace (integer), palette/tRNS handling; growable buffers (near). Optional gamma/color-management uses floats and is deferred without affecting core decode.
- **Risk:** libpng's API/ABI is sprawling and callback-driven, so a faithful drop-in is more surface than the decode logic itself; ancillary-chunk handling invites scope creep.

### safe-jpeg: libjpeg-ABI baseline decoder  `[near]`
- **Headline:** The most-attacked image format on earth, JPEG, decoded by an AI into a libjpeg-compatible library where the classic Huffman and upsampling overflow CVEs cannot be written.
- **Vs Rust wave:** Rust has fast JPEG decoders (jpeg-decoder, zune-jpeg), but they are crates rather than libjpeg.so drop-ins and zune leans on unsafe SIMD; the vast base of apps linking libjpeg/libjpeg-turbo cannot swap them in, whereas Whitefoot can emit the libjpeg ABI with zero unsafe and AI authorship.
- **Swap-in:** LD_PRELOAD over libjpeg.so/libturbojpeg for the decompress path (jpeg_read_scanlines, tjDecompress); libjpeg's ABI is the closest thing to a universal image-decode standard, though its state-machine API is large.
- **Verify:** Decode libjpeg-turbo's test images and compare RGB output (bit-identical when pinned to the same integer IDCT variant); run tjbench/tjunittest; fuzz-diff against libjpeg-turbo and replay heap-over-read PoCs such as CVE-2020-13790.
- **Gaps:** Baseline uses integer Huffman decode, integer dequant, libjpeg's own integer IDCT (jidctint), and fixed-point YCbCr-to-RGB, all integer/table (have + near tables). Progressive/arithmetic coding and any float IDCT path are deferred (far).
- **Risk:** Bit-exact parity hinges on matching a specific IDCT and upsampling variant; progressive JPEG is common in the wild and is a large second phase; the full drop-in has a big API surface.

### safe-stbi: stb_image link-compatible loader (BMP/TGA/GIF/PNG)  `[near]`
- **Headline:** An AI rewrote the stb_image loader embedded in thousands of games and tools so its long tail of parser overflows simply cannot exist.
- **Vs Rust wave:** Rust's image crate covers the formats but cannot be #include'd or linked as stbi_load, so the game/engine/tool ecosystem that copy-pastes stb_image.h has no path to it; Whitefoot emits a C library with the identical stbi_load/stbi_load_from_memory signatures as a literal link-time swap, zero unsafe, AI-authored.
- **Swap-in:** Build a C-ABI object exporting stbi_load* with matching signatures and channel semantics for a link-time replacement, plus LD_PRELOAD where the host links it dynamically. Scoped to the memory-safe integer formats.
- **Verify:** Diff decoded pixels against stb_image over its own test corpus and a broad image set; fuzz-diff per format; replay historical stb_image crashers.
- **Gaps:** BMP/TGA/GIF are self-contained integer decoders; PNG reuses the inflate kernel; needs growable buffers (near). Excludes HDR/float formats and PSD.
- **Risk:** stb_image has many lenient quirks and edge behaviors that callers silently depend on, so bit-identical-including-the-weird-cases is the hard part; format breadth is needed for convincing parity.

### safe-tt: TrueType sfnt parser + rasterizer  `[far]`
- **Headline:** The FreeType font-parsing 0-day class behind Chrome's CVE-2020-15999 is unrepresentable in this AI-written glyph engine.
- **Vs Rust wave:** Rust font stacks exist (read-fonts/swash, ab_glyph) and are promising, but none is a deployed libfreetype.so replacement, and the sfnt-table plus hinting-VM surface is exactly where the 0-days live; Whitefoot's parser is AI-authored with no unsafe in the table/offset math, so the embedded-PNG/color-bitmap overflow that was CVE-2020-15999 becomes a checked-bounds impossibility.
- **Swap-in:** Partial libfreetype.so ABI shim for the load-glyph/render path, or embed as a font sidecar in a specific app; the full FreeType ABI is very large, so this is a scoped drop-in, not a transparent total replacement.
- **Verify:** Parse and rasterize the Google Fonts corpus and diff glyph bitmaps against FreeType; replay the CVE-2020-15999 PoC font and other fuzzer-found crashers; fuzz-diff table parsing.
- **Gaps:** cmap/glyf/loca/CBDT table parsing is integer (have); 26.6 fixed-point rasterization is integer (have); the TrueType hinting bytecode interpreter is a large integer VM (buildable but sizable); CFF/Type2 and color/PNG-in-font are further scope. No floats required.
- **Risk:** Largest scope of the slate (hinting VM plus rasterizer bit-exactness), and a full ABI drop-in is impractical, so the realistic win is a safe font sidecar rather than a transparent libfreetype swap.

### safe-gif: libgif-ABI GIF/LZW decoder  `[near]`
- **Headline:** An AI wrote a drop-in giflib replacement in which the LZW decode overflows that have haunted GIF parsers for decades cannot occur.
- **Vs Rust wave:** image-rs decodes GIF in Rust, but again it is a crate, not a libgif.so (DGifOpen/DGifSlurp) drop-in for the C apps and image tools that link giflib; Whitefoot emits the small giflib ABI directly, with zero unsafe in the LZW loop, AI-authored.
- **Swap-in:** LD_PRELOAD/symlink over libgif.so; giflib's ABI is small enough (DGifOpen*/DGifSlurp/DGifCloseFile) to make a faithful drop-in realistic, unlike the sprawling image libraries.
- **Verify:** Decode a broad GIF corpus and compare frames against giflib; fuzz-diff; replay giflib crashers including the CVE-2019-15133 family and heap-overflow PoCs.
- **Gaps:** LZW code-table decode plus interlace handling is pure integer/table over byte buffers (have + near const tables); growable output buffer (near). No floats/strings.
- **Risk:** Small and clean, so the main risk is matching giflib's exact error and partial-frame behavior; multi-frame animation edge cases and disposal semantics need care to stay bit-identical.

### safe-der: ASN.1/DER X.509 certificate parser  `[near]`
- **Headline:** The TLS certificate-parsing memory bugs that have repeatedly rocked OpenSSL and GnuTLS are unrepresentable in this AI-written DER parser.
- **Vs Rust wave:** This is the most Rust-contested target on the list: rustls/webpki and the der/x509-parser crates already made this argument and are deployed. The honest differentiator is narrower, not novel: Whitefoot's parser is AI-authored with zero unsafe (some Rust ASN.1/crypto paths still use unsafe), and length/tag integer handling cannot silently wrap, so the OpenSSL negative-length/overflow lineage (CVE-2016-2108 family) is structurally excluded.
- **Swap-in:** Standalone C-ABI validating parser, or a BoringSSL/OpenSSL-embeddable d2i-shaped module for the cert-parse path. Weak as a universal drop-in because TLS libraries do not share one ABI, so realistic use is a hardened front-end that parses/validates structure before the crypto library.
- **Verify:** Parse a large corpus of real certificates (Certificate Transparency logs) and diff the parsed structure against OpenSSL; fuzz-diff DER decoding; replay archived OpenSSL/GnuTLS ASN.1 CVE PoCs and confirm clean rejects.
- **Gaps:** TLV/length/OID/tag parsing is pure bounded integer/byte work (have); big-integer key fields parse as byte buffers (have) but signature/crypto math (RSA/EC) is deliberately out of scope (far). No floats/strings.
- **Risk:** The most crowded-by-Rust target, so the story leans hardest on AI-authorship plus zero-unsafe rather than newness; parsing-only scope hardens the front door but still hands off to a C crypto core, bounding the end-to-end safety claim.

## Lens: dev-virality

### libz drop-in: memory-safe inflate/deflate (gzip)  `[near]`
- **Headline:** An AI wrote a memory-safe gzip you LD_PRELOAD under your web server and git clone, and it matches zlib-ng with zero unsafe code and none of the inflate CVE class.
- **Vs Rust wave:** The Rust rewrite already exists (zlib-rs) and is fast, but its speed rests on unsafe SIMD and unchecked 32KB-window indexing — the exact code where zlib's real CVEs live (CVE-2022-37434 inflate heap over-read, CVE-2018-25032 deflate corruption). Whitefoot's win is different in kind: that over-read class is unrepresentable with no writer-reachable unsafe, and the src/dst window disjointness is a checked first-class fact (the noalias rustc has historically had to disable), not an inference — so the safe build is also the fast build.
- **Swap-in:** ABI-compatible libz.so.1 exporting inflate/deflate/inflateInit2/etc. — LD_PRELOAD it under git, curl, rsync, nginx; plus drop-in gzip/gunzip/zcat CLIs. Package installs (npm), HTTP responses, and git object reads all sit on this one hot path.
- **Verify:** Bit-identical inflate output over Canterbury + Silesia corpora and zlib's own test suite; differential fuzz against stock zlib on Google's OSS-Fuzz zlib corpus; deflate output round-trips through reference zlib inflate. Decode is bit-exact by RFC 1951, so the strongest claim is fully verifiable.
- **Gaps:** Growable output buffer (near); thin I/O frame for streaming read/write (near); const Huffman/length tables (near). Deflate match-finder wants a fixed hash-chain buffer (have via P2 SoA pool). No strings, floats, or concurrency needed at all.
- **Risk:** Deflate COMPRESSION producing byte-identical output to zlib is hard (match-finder heuristics), so lead with inflate (decode) bit-exactness and claim ratio-parity, not byte-parity, for compress. zlib's C ABI is broad; enough of the surface must be implemented for a clean LD_PRELOAD under real programs.

### liblz4/snappy block codec (data-infra)  `[near]`
- **Headline:** The compression library inside Kafka, RocksDB, and ClickHouse, re-implemented by an AI: LZ4-speed block codec with zero unsafe and no possible wildcopy overflow.
- **Vs Rust wave:** Fast LZ4/Snappy in Rust either binds to the C library or uses unsafe wildcopy/wide-copy loops — the overlapping-copy step that is the textbook source of decompressor OOB writes. Whitefoot expresses the overlap copy as bounds-safe by construction while the noalias facts on the source/dest cursors keep the copy at C throughput, so 'no unsafe' and 'full speed' stop trading off. Nobody's Rust story is 'the AI wrote our storage-engine codec and it can't corrupt memory.'
- **Swap-in:** ABI-compatible liblz4.so / libsnappy.so linked or LD_PRELOAD'd under Kafka, RocksDB, ClickHouse, MongoDB (all pluggable-compressor); plus the lz4 CLI. Memory-to-memory block API means no I/O dependency for the core.
- **Verify:** Bit-identical decompress vs reference liblz4/snappy over each project's fuzzer corpus; compress→reference-decompress round-trip; differential fuzz on random blocks. Decode is bit-exact by spec.
- **Gaps:** Nearest of all the codecs: fixed-size block API needs only integers + byte buffers + a fixed hash table (all NOW); growable output buffer (near) only for the streaming/frame variant. No entropy stage, so no big table machinery. No strings/floats/concurrency.
- **Risk:** Decompress is trivially bit-exact; the honest caveat is compression RATIO parity, which depends on match-finder tuning — claim 'within a few percent of reference ratio at equal or better speed,' not byte-identical compressed output.

### zstd decompressor (modern default)  `[near]`
- **Headline:** zstd, re-implemented by an AI in a language where the buffer-overflow class that has hit compression libraries literally cannot be written — and decode speed holds.
- **Vs Rust wave:** zstd is the modern default (Linux kernel modules, Docker/OCI layers, Arch pacman, npm tarballs) and its fast decoders are C or unsafe-Rust intrinsics over FSE/Huffman tables and a sliding window. Whitefoot decodes the entropy and sequence stages with shaped-buffer no-bounds-check speed and zero writer-reachable unsafe; the window/table indexing that carries the OOB risk is checked, not asserted. The differentiator over the Rust wave is the authorship + provable absence of the corruption class, on the codec everyone is migrating to.
- **Swap-in:** ABI-compatible libzstd.so.1 (decode entry points) + zstd -d / zstd -t CLI; LD_PRELOAD under pacman, Docker, or any consumer of the frame format. Package-install and container-pull latency is felt directly.
- **Verify:** Bit-identical decode vs reference zstd over its published test vectors and decodecorpus; zstd -t integrity check; differential fuzz against the reference decoder. Decode output is bit-exact by RFC 8878.
- **Gaps:** Growable output + window buffers (near); FSE/Huffman entropy tables as const+dynamic buffers (near); thin I/O frame (near). Bigger state machine than gzip. No strings/floats/concurrency; dictionaries can be a later scope.
- **Risk:** Largest spec of the codec ideas (FSE + sequence decode + optional dictionaries) — scope to decode-only, no-dictionary first to keep it bounded. Compression parity is even harder than gzip's, so stay on the decode claim for the headline.

### simdjson-class JSON validator + minifier  `[near]`
- **Headline:** simdjson's GB/s, none of the unsafe: an AI-written JSON validator/minifier that hits multi-GB/s structural validation in pure checked code.
- **Vs Rust wave:** simd-json (Rust) is a near-direct port of C++ simdjson and is dense with unsafe, transmutes, and unchecked SIMD — GB/s parsing has always meant 'give up bounds safety.' Whitefoot reaches the same throughput from shaped-buffer bounds-check elimination plus noalias on the input/structural-index buffers, with zero unsafe, and the UTF-8 well-formedness check that JSON needs is a byte-level computation that requires no string/unicode subsystem at all. It re-runs simdjson's single most viral benchmark from the one angle the Rust port can't claim.
- **Swap-in:** Pipeline CLI: a GB/s json-validate (exit-code gate for CI/log ingestion) and a whitespace-stripping minifier that drops into build and log pipelines where jq . is used only to pass-through/validate. Also a library behind a CI validation step.
- **Verify:** Run JSONTestSuite (the canonical y_/n_/i_ conformance corpus) and accept exactly the valid set; byte-identical minify vs a reference minifier; differential fuzz against simdjson::validate on random and mutated documents.
- **Gaps:** Structural validation + minify + integer-number extraction: growable structural-index buffer (near), thin I/O frame (near), byte-level UTF-8 validation (NOW). FULL serde-style deserialize into number VALUES needs floats (far); a first-class string type would want unicode (far) — so scope the demo to validate/minify/integer-extract.
- **Risk:** Calling it a 'JSON parser' while float number-values are out (far) invites 'it doesn't fully parse' pushback next to simdjson. Must market precisely as validate+minify+structural, which is complete and useful for CI/log pipelines but narrower than a deserializer.

### Overflow-checked hashing at SIMD speed (xxhash/CRC32C)  `[near]`
- **Headline:** The thing Rust makes you choose between — overflow-checked arithmetic and SIMD throughput — an AI got both at once in a hashing/checksum library.
- **Vs Rust wave:** In Rust a checked/overflow-trapping reduction does not vectorize: the panic-ordering and loop-carried check dependency block it, so you pick speed (wrapping) or safety (checked). Whitefoot's checked associativity law (channel 3) licenses reassociating a trap/checked reduction into independent SIMD lanes anyway, and CRC's table lookups are no-bounds-check by shape — overflow-safe hashing at wrapping-hash speed, which Rust's arithmetic model can't express. (Honest scope: for plain wrapping xxh3 whose lanes are already independent, the edge narrows to no-unsafe-SIMD + authorship — so lead with CRC/checked-sum.)
- **Swap-in:** ABI-compatible libxxhash / crc32c linked or LD_PRELOAD'd under RocksDB, zstd, ccache/sccache, and content-addressed build caches (Bazel/Turborepo hash everything); plus xxhsum/cksum CLIs. Build-cache and dedup latency is felt on every incremental build.
- **Verify:** Bit-identical digests vs reference xxh3/CRC32C over SMHasher's vectors; SMHasher quality suite pass; differential against the reference on random inputs of every length/alignment.
- **Gaps:** Core is NOW: all-width integers + byte buffers. CRC needs a const table (near); streaming needs the thin I/O frame (near). No strings/floats/concurrency.
- **Risk:** The clean channel-3 win must be shown on a benchmark where Rust's checked path genuinely fails to vectorize (CRC / overflow-checked sum), or the differentiator reads as manufactured. Plain wrapping-hash throughput is already reachable in unsafe Rust, so the framing has to foreground the checked-arithmetic edge, not raw MB/s.

### Myers line-diff (git diff / GNU diff)  `[near]`
- **Headline:** An AI wrote git's diff hot path in a language with no buffer overflows and no unsafe, and the large-file diff stayed fast.
- **Vs Rust wave:** The LCS/edit-distance DP behind diff is a wall of index arithmetic over big matrices; Rust either eats a bounds check per access or reaches for unsafe get_unchecked to keep up. Whitefoot gives the DP arrays no-bounds-check via carried shape plus noalias on the two line buffers — full speed, zero unsafe — and the AI authored a notoriously fiddly algorithm correct-by-construction. The plain 'rewritten in Rust' story (the similar crate) doesn't carry the authorship or the provable-safety angle.
- **Swap-in:** Drop-in diff binary; or link/LD_PRELOAD an xdiff-shaped library into git so git diff / git blame get faster on large repos. Diff latency is felt in every code review and rebase.
- **Verify:** Apply-correctness: the emitted diff patches source→target exactly, checked over a large file corpus; differential vs GNU diff / git xdiff on edit-count minimality and on agreement for unambiguous cases; fuzz random edits.
- **Gaps:** Line splitting on newline bytes (NOW), line hashing (NOW, integers), growable + dynamically-sized DP buffers (near). Byte/line diff needs no unicode. Histogram/patience heuristics are more code but fully in-subset.
- **Risk:** Diff output is not uniquely defined, so you cannot claim byte-identical vs git — verification is 'valid, minimal-ish, patches correctly,' a weaker proof for a headline. Matching git's histogram heuristic exactly to avoid noisy diffs is extra work.

### grep -F fixed-string byte search (ripgrep's turf)  `[near]`
- **Headline:** grep -F, re-done by an AI with zero unsafe, matching GNU grep byte-for-byte at full memchr-class scan speed.
- **Vs Rust wave:** ripgrep is THE Rust rewrite poster child, and its scan speed comes from the memchr crate — famously unsafe-heavy SIMD skip loops. Whitefoot gets Boyer-Moore/skip-scan throughput with no writer-reachable unsafe, and the AI authored it. The point is precisely to plant a flag on the Rust wave's home turf with the one claim ripgrep can't make: provably no memory-unsafety, machine-written.
- **Swap-in:** Drop-in grep -F-compatible binary usable in existing shell/log pipelines; line-oriented byte matching over stdin or files.
- **Verify:** Identical match set and line/byte offsets vs GNU grep -F over a large corpus; differential fuzz on random needles and haystacks including overlaps and boundary cases.
- **Gaps:** Fixed-string / byte-pattern search: byte buffers (NOW) + thin I/O frame (near). Full regex is FAR (needs an engine), unicode-aware matching and case-folding beyond ASCII are FAR — so the honest NOW/near scope is literal and simple byte-class search only.
- **Risk:** Highest-blowback idea: going at ripgrep directly while the honest scope (fixed-string, no full/unicode regex) is a fraction of what ripgrep does invites 'that's not really a grep.' High viral ceiling, high reputational downside — frame it explicitly as the fixed-string fast path, not a regex engine.

### AI-authored minifier / bundler step (esbuild/swc turf) — aspirational  `[far]`
- **Headline:** The esbuild story nobody in the Rust wave can copy: a JS/CSS minifier the AI wrote end-to-end, provably memory-safe with zero unsafe.
- **Vs Rust wave:** Toolchain speed is the most viral dev-tool genre (esbuild, swc, bun, oxc, biome), but it is now SATURATED — 'faster minifier rewritten in Rust/Go/Zig' is no longer novel. Include as the far-rung aspirational anchor, not a near-term deliverable; overreaching here before strings/unicode land would produce a toy that invites dismissal.

## Lens: ubiquity-infra

### LZ4 block decompressor (drop-in liblz4)  `[near]`
- **Headline:** An AI wrote a bit-identical, memory-safe LZ4_decompress_safe with zero unsafe — same speed, and the heap-overflow class behind CVE-2019-17543 / CVE-2021-3520 is now unrepresentable.
- **Vs Rust wave:** Rust DOES have lz4_flex, but it ships an unsafe fast-decode path for speed (the 'safe' pure path is slower) and a human wrote it. Whitefoot's wedge is categorical, not incremental: there is no unsafe escape hatch anywhere, so the safe path IS the only path AND the fast one — the checker's per-access bounds/alias facts feed the optimizer, and the AI authored every byte. Rust literally cannot say 'no unsafe in the hot copy loop.'
- **Swap-in:** ABI-compatible liblz4.so exporting LZ4_decompress_safe(const char* src, char* dst, int srcSize, int dstCapacity) and LZ4_decompress_safe_partial. Caller-provided dstCapacity fits the non-growable buffer<u8> model exactly. Vendored kernel into Kafka / RocksDB / MongoDB / kernel-userspace zram consumers, or LD_PRELOAD.
- **Verify:** Fuzz-diff every output byte against reference liblz4 on the lz4 test corpus + Google's lz4 OSS-Fuzz corpus; bit-identical decompress of the Silesia corpus; round-trip against the C compressor. Any divergence is a hard fail.
- **Gaps:** The match-copy does an intentional OVERLAPPING self-copy inside one buffer (match distance < match length is deliberate in LZ4). That stresses indexed read+write on a single &uniq buffer<u8> — in-subset in principle (one owner, indexed access) but not yet demonstrated; must be byte-at-a-time, not a bulk memmove. No growable buffer needed since dstCapacity is caller-given.
- **Risk:** Byte-at-a-time overlapping copy can lose to liblz4's word-at-a-time wild-copy; the optimizer must recover it or the '~same speed' claim weakens. This is the swap-in-clean, CVE-shape-perfect leaf — strongest all-around near-term pick.

### zlib inflate (DEFLATE decompressor, drop-in libz)  `[near]`
- **Headline:** An AI re-implemented zlib's inflate with zero unsafe — decompresses gzip/zlib/raw DEFLATE bit-for-bit at C speed, and CVE-2022-37434-class heap over-reads cannot be written.

The fast shapes, computed-goto threading and musttail+`preserve_none` handler chains, are exactly what wasm3 hand-writes in C and Rust cannot. Whitefoot's closed-world exhaustive `match` licenses the compiler to tail-duplicate dispatch, so the naive source becomes the fast shape.
- **Swap-in:** ABI-compatible interpreter core dropped where wasm3's `m3_*` executes, or a Pike-VM/regex-bytecode core swapped into the `regex` engine's exec loop; run the incumbent's suite unchanged.
- **Verify:** The WebAssembly spec test suite (or the `regex` crate corpus) for behavioral parity; perf-counter branch-misprediction rate plus ns/op vs Wasmi (Rust switch) and wasm3 (C hand-threaded); assembly diff showing tail-duplicated `indirectbr`/musttail chains rather than one shared branch.
- **Gaps:** Carded as candidate 'channel 4,' NOT built. Needs musttail+`preserve_none` emission in the backend, const-array/table codegen for opcode tables (a near-term gap), and no strings/floats limits the target to integer i32/i64 bytecode (a wasm numeric subset fits; full Lua does not).
- **Risk:** LLVM's SimplifyCFG can re-merge the tail-duplicated dispatch blocks back into one shared branch (the reason GCC ships `-fno-crossjumping`); if the backend can't hold the duplication, the structural win collapses to parity. And it's parity-with-C, not a win over C, so the whole story is 'Rust can't, we can,' narrower than 'fastest.'

### Zero-unsafe, AI-authored decoder: memory-corruption CVE class unrepresentable  `[near]`
- **Headline:** An AI reimplemented a QOI/PNG decoder with no line a human has to trust: the language has no `unsafe`, so the entire memory-corruption CVE class is a compile-time impossibility.
- **Vs Rust wave:** The Rust rewrite wave (uutils, sudo-rs) still ships `unsafe` for perf/FFI/convenience, and an AI writer can reach for `unsafe` to silence the borrow checker, the cheat path that turns a red program green. Whitefoot has no writer-reachable escape hatch, so 'the AI cannot have introduced memory corruption' is a checkable property of the accepted artifact (grep: zero unsafe lines), not a promise about review diligence, and unlike safe Rust in release it also closes silent integer overflow so length/offset math can't wrap into an OOB.
- **Swap-in:** ABI-compatible C object exposing the incumbent symbol (`qoi_decode`, `stbi_load` for a format subset, `base64_decode`, a protobuf varint reader); LD_PRELOAD or link into the incumbent's test harness and differential-fuzz against it.
- **Verify:** The incumbent's own KATs/test images for bit-identical output; libFuzzer/AFL differential fuzzing whitefoot-vs-C on a shared corpus (same output on valid input, both reject invalid) with ASan on the C side lighting up where Whitefoot is clean by construction; trust surface = 0 `unsafe`.
- **Gaps:** Needs growable output buffers and the thin read/write/open I/O frame (both named near-term gaps) plus const tables. Byte-only formats (QOI, base64, varint, TAR headers, UTF-8 validate) fit; anything needing strings/floats (JPEG, full PNG with zlib) is far.
- **Risk:** This is the closest to the plain 'rewrite in a safe language' story the strategy warns Rust already owns; `#![forbid(unsafe_code)]` safe-Rust is a real competitor for the narrow claim. The honest differentiator is the STACK, no-unsafe AND per-op defined overflow (which safe release Rust lacks) AND AI-authored with a mechanical trust surface, not any single leg; pitched as just 'no unsafe,' a skeptic deflates it.

### Overflow behavior on the operation, not the build mode  `[now]`
- **Headline:** The integer-overflow CVE class, closed per operation: an AI wrote allocation-size math that traps on overflow in every build, and the check costs nothing when the compiler can discharge it.
- **Vs Rust wave:** Rust ties overflow checking to build mode (`overflow-checks` on in debug, off in release by default), so shipped release binaries silently wrap; debug and release are literally different programs and you profile a build you didn't test. Closing it in Rust means remembering `checked_*`/`saturating_*` by hand at every site; Whitefoot makes the mode mandatory and per-op (`iadd.trap`/`.checked`/`.sat`/`.wrap`), identical in every build, and deletes the check wherever it proves overflow impossible.
- **Swap-in:** ABI-compatible object for a size/offset calculation or integer parser (the classic `count * elem_size` heap-overflow pattern behind Stagefright-class CVEs); link into the incumbent and fuzz-diff.
- **Verify:** Identical results vs the incumbent on in-range inputs; on crafted overflowing inputs Whitefoot traps where C/release-Rust silently wrap into a corrupt size; IR/asm shows the bounded-loop checks folded away to a constant (README example 3, already measured), proving 'checked but free.'
- **Gaps:** Minimal for the arithmetic core (numeric modes plus discharge are built, `checked_integer_parser` is a reference kernel); a full allocation-size drop-in additionally wants the alloc/I-O frame (near).
- **Risk:** 'Rust has `checked_add`' is the immediate rebuttal; the delta is mandatory/per-op/build-invariant/free-when-discharged, not a novel operation, so the headline must sell the DEFAULT and the FREE, not the existence of checked math. Modest ceiling alone; strongest as a supporting pillar under the CVE-class story.

### Effect-typed pure calls: LTO-class hoisting without LTO, across an opaque boundary  `[now]`
- **Headline:** A function in a shared library the caller optimizes like a constant, hoisted out of the hot loop across the .so boundary with no LTO, because its signature says `pure`.
- **Vs Rust wave:** Effects live nowhere in Rust/C signatures, so at a real `dlopen`/dylib boundary, or under ELF semantic interposition where even LTO can't reach, the caller must assume the callee touches global state and cannot hoist, CSE, or reorder the call. Rust recovers this only with whole-program fat LTO (inlining the callee); Whitefoot emits `readnone`/`readonly` from the effect row so the call optimizes like a value while the callee stays opaque and un-inlined.
- **Swap-in:** An ABI-compatible plugin `.dylib` (image/codec/hash/validate plugin, a PostgreSQL extension function, a Lua/Python C-extension hot path) whose exported pure function the host calls in a loop; measure calls elided.
- **Verify:** Measured O(n)->O(1) at the opaque boundary, tying fat-LTO Rust without LTO (`experiments/effect-attrs-channel`, pinned in `make check`); IR shows the `readnone`/`readonly` attribute; ablating the attribute (a metadata-only diff) brings the repeated work back.
- **Gaps:** In-process the win collapses to parity, safe Rust plus fat LTO reaches the same ceiling, so the durable delta is specifically the opaque boundary (plugins, `dlopen`, interposition) where LTO cannot inline; the `-fno-semantic-interposition` nuance applies on the C side.
- **Risk:** The project's own skeptical audit already flagged that this 'collapses to automation' whenever LTO is available; if a demo isn't across a genuinely opaque boundary it's just parity, and 'optimize across a .so boundary' is abstract to normal developers, with lower visceral pull than the crypto or CVE-class stories.

### Fearless disjoint parallel scatter with no `unsafe`  `[far]`
- **Headline:** The one parallel pattern safe Rust can't reach without `unsafe`: a provably-disjoint scatter an AI wrote safely and the compiler vectorizes.
- **Vs Rust wave:** `out[perm[i]] = f(in[i])` is safe and parallelizable exactly when `perm` is injective, but Rust's type system can't express injectivity, so safe Rust either pays redundant bounds/aliasing traffic or drops to a single `unsafe` store to close the gap. Whitefoot's ownership/region provenance is the natural place to carry disjointness as a checked fact, keeping the fast store safe.
- **Swap-in:** ABI-compatible kernel for a permutation/gather-scatter step (radix-sort scatter, histogram/CSR assembly, a graph relabeling pass) linked into the incumbent's pipeline; diff outputs and count the eliminated redundant traffic.
- **Verify:** Identical output vs the incumbent on random permutations; on a non-injective permutation the guard/law refutes it (no silent corruption); measured 1.13-1.51x memory-bound over best-effort safe Rust in the scatter study, with one `unsafe` store shown to be what safe Rust needs to match.
- **Gaps:** The load-bearing injectivity fact is NOT yet in Whitefoot's effect/region vocabulary; the in-house study is explicit that expressing it needs index-parameterized regions or a law channel, the fine machinery that broke DPJ's annotation budget. A per-dispatch O(n) verification guard erases the memory-bound win unless the proof is static or amortized over many kernel runs.
- **Risk:** The project's own adversarial audit came out largely NEGATIVE on auto-parallelism as a pillar: the residual is narrow (1.1-1.5x, not orders of magnitude), DPJ (Whitefoot's near-twin) got near-zero uptake, and rayon's `.par_iter()` already ties the task-level ceiling. This is honest and real but a niche optimization, not a project-defining headline, and must be pitched as such.
